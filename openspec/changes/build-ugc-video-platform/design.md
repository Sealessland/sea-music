## Context

仓库当前没有业务代码或既有规格。项目的首要目标是形成一套能面向 B 站及同类内容平台后端岗位讲清楚、跑得起来、测得出来的 UGC 视频核心链路，而不是用大量空壳服务模拟“大规模”。主要使用者是开发者、面试评审者和本地演示者；系统需要在一台开发机上完整启动，同时保留向生产形态演进的边界。

核心数据流如下：

```text
Client -> HTTP API -> PostgreSQL (business rows + outbox)
              |  \-> Redis (limits, cache, ranking)
              \----> S3-compatible storage (direct upload/playback grants)

Outbox dispatcher -> Kafka-compatible broker -> media/aggregate/ranking consumers
                                                |       |        |
                                                v       v        v
                                              FFmpeg  Postgres  Redis

All components -> OpenTelemetry -> Prometheus / tracing / structured logs
```

## Goals / Non-Goals

**Goals:**

- 完成从注册、上传、处理、发布、发现、播放元数据到互动的真实业务闭环。
- 将幂等、一致性、热点、缓存、异步任务和故障恢复落实为可执行测试与可观测证据。
- 让单机开发环境简单可复现，并允许未来按模块提取独立服务。
- 形成适合简历陈述的工程结果：明确基线、压测条件、瓶颈和优化前后对比。

**Non-Goals:**

- 不自研 CDN、对象存储、编解码器、全文搜索引擎或机器学习推荐模型。
- 不首版实现直播、私信、支付、广告、版权 DRM、多地域容灾或完整内容审核平台。
- 不宣称百万 QPS、海量数据或生产可用性；只报告在明确硬件和数据规模下实测的结果。
- 不为每个领域模块预先部署独立微服务。

## Decisions

### 1. Go 模块化单体 API + 独立 Worker

代码按 `identity`、`video`、`social`、`discovery`、`events` 和 `platform` 领域模块组织。模块拥有自己的应用服务、存储接口和数据表，禁止跨模块直接访问内部实现；API 进程负责同步请求，Worker 进程承载转码、聚合、排名和事件投递。

选择该形态是因为个人项目更需要完整性和深度。它保留进程边界、消息契约和数据所有权，后续可依据扩缩容、故障隔离或团队所有权提取服务。备选的“全微服务”会显著增加部署和分布式事务成本，却不会自动证明系统能力；单进程无 Worker 则无法真实展示异步可靠性。

### 2. 外部 REST/JSON API，契约先行

对外使用版本化 REST/JSON，并维护 OpenAPI 契约、统一错误模型、请求标识和 opaque cursor。模块内部先使用 Go 接口和类型；只有提取进程后才引入 RPC 契约。

相比首版同时引入 gRPC，REST 更利于演示、调试和前端接入。服务拆分时可在内部增加 Protobuf/gRPC，而无需污染当前领域边界。

### 3. PostgreSQL 是权威数据源，Redis 只保存可重建状态

用户、视频状态、互动关系、评论、事件和任务尝试保存在 PostgreSQL。唯一约束、外键、版本列和事务负责不变量。Redis 用于限流、热点缓存、公开计数和排名集合；缓存丢失不得造成权威关系丢失，并由回源或重建恢复。

选择 PostgreSQL 而非多种专用数据库，可把一致性问题讲深并降低本地复杂度。后续只有在查询证据表明必要时才为弹幕或检索引入新存储。

### 4. S3 直传 + FFmpeg 真实处理

API 发放短时上传授权，客户端直传 S3 兼容存储；finalize 时验证对象元数据并创建处理任务。Worker 使用 `ffprobe` 校验媒体并用 `ffmpeg` 生成至少一种可播放 rendition 和封面，产物键由资产 ID 与配置版本确定，从而支持幂等重试。本地使用 SeaweedFS 的 S3 API，接口保持兼容托管对象存储。最初考虑 MinIO，但其社区版从 2026 年起不再提供维护中的预编译镜像，不适合作为可复现开发环境的新依赖。

不通过 API 代理大文件，以避免应用内存和带宽成为伪造的核心瓶颈；不使用“sleep 后标记转码成功”的模拟任务，因为它无法验证租约、产物幂等和失败恢复。

### 5. 事务 Outbox + Kafka 兼容消息流

业务事务同时写 Outbox。Dispatcher 使用带锁批量领取、稳定事件 ID 和 broker ack 发布。消费者以事件 ID 建立 Inbox/唯一性边界，在成功提交副作用后确认消息；重试耗尽进入死信主题并保留可审计重放入口。本地优先使用单节点 Kafka 兼容实现以降低资源占用，协议不绑定具体发行版。

直接“写数据库后发消息”存在双写窗口；分布式事务对本项目过重。Outbox 能明确展示至少一次投递和幂等消费的工程取舍。

### 6. 投稿采用版本化状态机，任务采用租约

视频状态转换集中定义，通过版本列进行 compare-and-swap。媒体任务记录配置版本、尝试次数、租约持有者、租约截止时间、最后错误和产物清单。消费者重复、进程崩溃或旧任务晚到时，不得回退已经推进的状态。

这比散落的状态字段更新更容易验证非法转换和竞态，也为后台人工恢复提供审计信息。

### 7. 互动关系强一致，计数和榜单最终一致

点赞、收藏、关注以 PostgreSQL 唯一关系为准；请求携带幂等键或利用自然唯一键安全重试。关系变更产生事件，异步更新 Redis/持久化聚合。定时 reconciliation 从权威关系修复计数漂移。热门榜单以去重事件、时间窗、权重和衰减计算，周期性保存快照。

将“是否点赞”和“点赞总数”分开，可让用户自身操作立即正确，同时避免高频计数锁争用。所有读响应需要标明或容忍计数的最终一致性。

### 8. 发现链路从可解释规则开始

关注流首版采用 fan-out-on-read：按关注集合查询已发布视频并使用 `(published_at, id)` 游标。热门与推荐从异步维护的候选集读取，在响应前回查可见性；推荐由关注、分类、新鲜度和热度组成，并返回 reason code。

相比伪造复杂推荐模型，这一方案可测、可解释。若关注数和读放大成为实测瓶颈，再引入混合 fan-out 或预计算收件箱。

### 9. 可观测性和验证是产品能力

所有入口、SQL、缓存、消息和媒体任务接入 OpenTelemetry；指标采用 RED/USE 思路并限制标签基数。容器编排包含 PostgreSQL、Redis、对象存储、消息代理和观测组件。集成测试使用真实依赖；单元测试只在领域边界使用 fakes。负载测试保留数据生成方式、场景配置、环境和原始结果。

验收优先看端到端状态和遥测证据，不能用被改弱的断言、mock broker 或自报吞吐替代核心路径。

## Risks / Trade-offs

- [本地基础设施较重，降低新用户启动成功率] → 提供分层 profile、固定镜像版本、健康检查和一条 bootstrap/verify 命令；核心验收 profile 仍使用真实依赖。
- [FFmpeg 处理不受信任媒体可能耗尽 CPU/磁盘] → 限制输入大小、时长、格式、并发和超时，将 Worker 与 API 分离并使用临时目录配额。
- [Outbox 清理或分区处理不当导致表膨胀] → 建立可观测 backlog、批量归档/清理策略，并用索引只扫描待投递记录。
- [异步计数短暂不一致影响体验] → 用户关系查询走权威存储，响应支持乐观 UI，定时 reconciliation 并暴露漂移指标。
- [fan-out-on-read 对高关注用户产生读放大] → 设置查询预算和降级，收集基数与延迟；达到可测阈值后演进混合 fan-out。
- [模块化单体仍可能发生边界侵蚀] → 通过包可见性、依赖规则测试和独立 repository 接口阻止跨模块数据访问。
- [项目范围仍偏大] → 按任务里程碑先交付投稿闭环，再交付互动、发现和工程强化；每个里程碑都必须独立可演示。

## Migration Plan

1. 建立工程骨架、契约、迁移系统和本地依赖，完成就绪验证。
2. 交付身份与投稿状态机，打通对象上传、真实处理和发布的最小纵切。
3. 引入 Outbox/消息消费，将处理和聚合从同步路径迁到 Worker；切换前保留可回滚的同步开发开关，但正式验收只走异步路径。
4. 增加互动、计数修复、关注流、热门和推荐召回。
5. 完成遥测、故障演练、安全检查和负载基线，冻结一套可复现演示数据。

数据库变更使用前向兼容迁移；部署先迁移再滚动进程。每阶段回滚仅回滚应用版本，不自动执行破坏性数据库 down migration；新表/列在确认无旧版本使用后另行清理。

## Open Questions

- 实现开始前确认候选人的主投语言是否必须为 Java；若是，保留领域、数据和事件设计，将 Go 技术实现替换为 Java/Spring Boot，不改变产品形态。
- 首版 UI 只需 API 文档和最小演示客户端，还是需要一个可公开访问的完整前端；这不阻塞后端纵切实现。
- 本地 Kafka 兼容发行版和观测 UI 的具体选择将在工程初始化时依据开发机资源与许可证固定。
